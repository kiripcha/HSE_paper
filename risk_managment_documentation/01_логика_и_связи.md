# Логика проекта и связи между модулями

Файл описывает, как устроен пакет `crypto_risk` изнутри: какие у него модули,
как они связаны, что подаётся на вход каждого и что возвращается на выход.
В сочетании с файлом «Подробная документация» этого достаточно, чтобы
ориентироваться в проекте без чтения исходников.

---

## 1. Назначение пакета

`crypto_risk` — модуль рыночного риск-менеджмента, встраиваемый в торговую
систему. Принимает на вход исторические котировки криптовалют, оценивает
рыночный риск портфеля разными методами, проверяет корректность моделей,
строит оптимальные веса портфеля, учитывает риск ликвидности и издержки
исполнения, проводит стрессовые проверки и адаптивно управляет размером
позиции. Все компоненты построены так, чтобы быть устойчивыми к ошибкам входа
и совместимыми друг с другом через единый внутренний контракт типов.

Принципиальное правило: «тонкие» модули не знают друг о друге, исключения —
несколько ленивых внутренних импортов. Оркестратор `engine.py` — единственное
место, где модули взаимодействуют.

## 3. Сквозной поток данных

Типичный сценарий «оценка риска текущего портфеля» проходит через следующие
этапы:

1. **Сбор котировок.** Загрузчик опрашивает несколько источников и собирает
   панель цен закрытия. Каждый источник возвращает временной ряд по одному
   активу; загрузчик объединяет их в одну таблицу и кэширует.
2. **Преобразование в доходности.** Цены переводятся в логарифмические
   доходности. Все последующие расчёты работают именно с ними.
3. **Оценка моделей волатильности.** На временной ряд одного актива
   подгоняется модель условной волатильности. Возвращается прогноз и сама
   условная σ во времени.
4. **Оценка мер риска.** На основе доходностей и при необходимости подогнанной
   модели волатильности рассчитываются VaR и ожидаемые потери разными
   методами. Каждый метод возвращает структуру с двумя числами и метаданными.
5. **Проверка моделей.** Скользящие оценки риска сравниваются с фактическими
   доходностями; рассчитываются формальные критерии корректности.
6. **Оптимизация портфеля.** На основе оценок матрицы ковариаций (выборочной,
   сжатой или динамической) находятся оптимальные веса портфеля при заданных
   ограничениях.
7. **Анализ ликвидности и стрессов.** К рассчитанному портфелю применяются
   модели риска ликвидности, стрессовые сценарии и обратное стрессовое
   тестирование.
8. **Адаптивный контроль.** На основе всех собранных оценок принимается
   решение о допустимом размере очередной заявки.

Все эти шаги собраны воедино в классе `RiskEngine`, который последовательно
вызывает соответствующие функции «тонких» модулей.

## 4. Контракты модулей

Для каждого модуля приводятся: назначение, перечень входных и выходных типов
данных, зависимости от других модулей и краткая логика.

### 4.1 config.py — параметры по умолчанию

| | |
|---|---|
| Назначение | Хранит все «магические числа» проекта в одном месте. |
| Вход | Нет. |
| Выход | Класс `RiskConfig` (вселенная, уровни доверия, горизонты, ставки, лимиты), константы (`RANDOM_SEED`, `TRADING_DAYS_YEAR`, `DEFAULT_UNIVERSE`), функция `to_symbol`. |
| Зависит от | Стандартной библиотеки. |
| Используется в | Каждом модуле проекта. |
| Логика | Простой контейнер. Изменение значения здесь меняет поведение всей системы без переписывания других модулей. |

### 4.2 data/sources.py — простой загрузчик

| | |
|---|---|
| Назначение | Загрузка биржевых свечей и стакана с одной биржи (через `ccxt`) и детерминированный генератор синтетических данных. |
| Вход | Название актива, диапазон дат, шаг свечи. |
| Выход | Таблица OHLCV с временным индексом или ряд цен закрытия. Для стакана — словарь с массивами заявок и сводными показателями. |
| Зависит от | `config.py`, библиотеки `ccxt` (опционально). |
| Используется в | `data/parsers.py`, `engine.py`. |
| Логика | Постраничная выгрузка с уважением к ограничениям биржи. При отсутствии сети — переход на синтетический генератор, который воспроизводит стилизованные факты крипты. |

### 4.3 data/parsers.py — мультиисточниковый сборщик

| | |
|---|---|
| Назначение | Загрузка длинной истории по нескольким активам из нескольких источников с дозаполнением пропусков. |
| Вход | Список активов, диапазон дат, режим работы (с сетью или оффлайн). |
| Выход | Таблица цен закрытия (строки — даты, колонки — активы). Атрибут `source_used` — что именно использовано. Метод `coverage_report` — отчёт по источникам. |
| Зависит от | `data/sources.py`, библиотек `requests`, `pandas`, `numpy`. |
| Используется в | `engine.py`. |
| Логика | Источники опрашиваются по приоритету. Первый, вернувший непустой ряд, становится основным. Остальные дозаполняют пропуски. Результат кэшируется в CSV. |

### 4.4 volatility.py — модели волатильности

| | |
|---|---|
| Назначение | Все способы измерения и прогноза условной волатильности. |
| Вход | Ряд логарифмических доходностей или таблица OHLC одного актива. |
| Выход | Ряд оценок σ во времени, прогноз σ на горизонт, объект подогнанной модели. |
| Зависит от | `config.py`, библиотеки `arch` (опционально). |
| Используется в | `var_es.py`, `evt.py`, `engine.py`. |
| Логика | EWMA — рекуррентная формула. GARCH — обёртка над `arch`. Диапазонные оценки — простые формулы по OHLC. HAR-RV — обычная регрессия на дневную, недельную и месячную реализованную дисперсию. |

### 4.5 var_es.py — VaR и ожидаемые потери

| | |
|---|---|
| Назначение | Расчёт мер риска несколькими методами. |
| Вход | Ряд доходностей (для большинства методов), параметры распределения, объект модели волатильности (для условных методов), веса портфеля и ковариация (для метода Монте-Карло). |
| Выход | Структура `RiskEstimate(var, es, method, horizon, ...)` для одного метода или таблица сравнения нескольких методов. Для декомпозиции — таблица вкладов активов. |
| Зависит от | `config.py`, `volatility.py`, ленивый импорт `evt.py`. |
| Используется в | `engine.py`, `backtesting.py` (через скользящие оценки). |
| Логика | Каждый метод реализован одной функцией с одинаковой сигнатурой. Соглашение: VaR и ожидаемые потери возвращаются как положительные числа, выраженные в долях капитала. |

### 4.6 evt.py — теория экстремальных значений

| | |
|---|---|
| Назначение | Оценка хвостовых мер риска через метод превышений над порогом. |
| Вход | Ряд доходностей. Опционально — объект GARCH-модели для условного варианта. |
| Выход | Подогнанные параметры обобщённого распределения Парето, оценка VaR и ожидаемых потерь, индекс хвоста Хилла. |
| Зависит от | `config.py`, `var_es.py` (для возврата структуры `RiskEstimate`). |
| Используется в | `var_es.compare_methods` (через ленивый импорт), `engine.py`. |
| Логика | Выбор порога по эмпирическому квантилю, MLE-оценка параметров. Условный вариант — фит GARCH, перенос модели на стандартизованные остатки. |

### 4.7 covariance.py — устойчивые оценки ковариационной матрицы

| | |
|---|---|
| Назначение | Снижение шума выборочной ковариации и динамические корреляции. |
| Вход | Таблица доходностей нескольких активов. |
| Выход | Квадратная матрица ковариаций. Для сжатия — также интенсивность сжатия. Для модели динамических корреляций — структура с условными матрицами и параметрами. |
| Зависит от | `config.py`, библиотеки `scikit-learn`, опционально `arch`. |
| Используется в | `portfolio.py`, `engine.py`. |
| Логика | Сжатие реализовано через готовые функции `scikit-learn` или вручную для варианта с постоянной корреляцией. Очистка по теории случайных матриц — спектральное разложение и замена шумовых собственных значений. Динамическая модель — двухшаговая. |

### 4.8 backtesting.py — обратная проверка моделей

| | |
|---|---|
| Назначение | Формальная проверка корректности моделей VaR и ожидаемых потерь. |
| Вход | Ряд реализованных доходностей и параллельный ряд оценок риска (или для ES — три параллельных ряда). |
| Выход | Структура `BacktestResult(name, statistic, p_value, reject_h0, detail)` для одного теста. Для сводного прогона — таблица результатов. |
| Зависит от | Только `config.py`, библиотек `scipy`, `numpy`. |
| Используется в | `engine.py` (через метод `backtest_var`). |
| Логика | Каждый тест реализован отдельной функцией с предсказуемым возвратом. Сводный прогон собирает их в таблицу. |

### 4.9 portfolio.py — оптимизация портфеля

| | |
|---|---|
| Назначение | Построение оптимальных весов портфеля при разных ограничениях. |
| Вход | Вектор ожидаемых доходностей и ковариационная матрица. Опционально — режим ограничений и пределы. |
| Выход | Структура `FrontierPoint(weights, ret, vol, sharpe)` для одного портфеля. Для границы — список таких структур. Для бет — структура `BetaEstimates`. |
| Зависит от | `config.py`, библиотеки `cvxpy` (опционально). |
| Используется в | `engine.py`. |
| Логика | Решение через `cvxpy` для constrained QP и аналитика для неконстрейнт-вариантов. Наиболее рискованный портфель решается через вершины симплекса. |

### 4.10 liquidity.py — риск ликвидности и исполнение

| | |
|---|---|
| Назначение | Учёт издержек исполнения и оптимальная стратегия закрытия позиции. |
| Вход | Ценовой VaR, статистики спреда, размер заявки и дневной оборот, данные по исполнению. |
| Выход | Структуры `LVaRResult`, `ImplementationShortfall`, `ExecutionSchedule` с разложением издержек и оптимальной траекторией. |
| Зависит от | `config.py`. |
| Используется в | `engine.py` (через методы риск-отчёта и предторгового контроля). |
| Логика | Все формулы реализованы напрямую. Для оптимальной стратегии — замкнутое решение с косинусом гиперболическим. |

### 4.11 dependence.py — копулы и хвостовая зависимость

| | |
|---|---|
| Назначение | Моделирование совместного распределения активов отдельно от маргиналов. |
| Вход | Таблица доходностей. |
| Выход | Объект подогнанной копулы (гауссова или стьюдентовская), матрица хвостовой зависимости, симулированные значения портфеля. |
| Зависит от | `config.py`, `var_es.py` (для функции «ближайшая положительно полуопределённая матрица»). |
| Используется в | `engine.py` (опциональная компонента риск-отчёта). |
| Логика | Псевдо-наблюдения через ранговое преобразование. Параметры копулы Стьюдента — MLE по сетке. Симуляция через стандартный приём: нормальные скоры, делённые на корень из хи-квадрат. |

### 4.12 stress.py — стрессовое тестирование

| | |
|---|---|
| Назначение | Оценка убытков в кризисных сценариях. |
| Вход | Таблица доходностей, веса портфеля, ожидаемые доходности и ковариация. |
| Выход | Структура `StressResult(name, pnl, detail)` для одного сценария; для набора сценариев — список или словарь таких структур. |
| Зависит от | `config.py`. |
| Используется в | `engine.py`. |
| Логика | Исторические сценарии — простая выборка худших периодов. Корреляционный стресс — пересчёт волатильности на новой корреляционной матрице. Наихудший случай и обратный стресс-тест — замкнутое решение через эллипсоид махаланобиса. |

### 4.13 controls.py — адаптивный сайзинг

| | |
|---|---|
| Назначение | Перевод оценок риска в размер позиции. |
| Вход | Прогноз σ, текущая просадка, ожидаемая доходность и ковариация. |
| Выход | Плечо, веса с целевой σ, оптимальная по Келли доля, множитель экспозиции в зависимости от просадки. |
| Зависит от | `config.py`, `volatility.py` (для дефолтного прогноза). |
| Используется в | `engine.py`. |
| Логика | Все функции — простые формулы. Объект `VolForecaster` инкапсулирует прогноз σ с возможностью подключения внешней модели глубокого обучения. |

### 4.14 engine.py — оркестратор

| | |
|---|---|
| Назначение | Единый интерфейс всех функций модуля для торговой системы. |
| Вход | Конфигурация (объект `RiskConfig`), вызовы методов с указанием весов, методов и параметров. |
| Выход | Структуры `RiskReport`, `PreTradeDecision` или непосредственные оценки конкретных мер. |
| Зависит от | Всех остальных модулей пакета. |
| Используется в | Внешний интерфейс. |
| Логика | Содержит состояние (загруженные котировки, кэш подогнанных моделей) и предоставляет высокоуровневые методы, которые внутри вызывают функции «тонких» модулей в нужной последовательности. |

## 5. Согласование форматов данных

Чтобы модули могли свободно сочетаться, в проекте принят небольшой набор
согласований по форматам.

* Цены — таблица `pandas.DataFrame` с временным индексом в UTC и колонками,
  соответствующими тикерам активов.
* Доходности — такая же таблица, но содержащая логарифмические приращения.
  Допускаются пропуски в начале периода для активов, появившихся позднее.
* Веса портфеля — словарь `{тикер: доля}` либо одномерный массив `numpy`.
  Сумма долей для длинных портфелей должна равняться единице.
* Ковариационная матрица — двумерный массив `numpy`, симметричный и
  положительно полуопределённый.
* Меры риска — объект класса `RiskEstimate` с полями `var`, `es`, `method`,
  `horizon`, `var_alpha`, `es_alpha` и опциональным словарём `extra`.
* Результаты обратной проверки — объект класса `BacktestResult` с полями
  `name`, `statistic`, `p_value`, `reject_h0`, `detail`.
* Результаты стрессовых проверок — объект класса `StressResult` с полями
  `name`, `pnl`, `detail`.

Эти соглашения обеспечивают возможность подменять одну реализацию на другую
без перенастройки остальной системы.

## 6. Пример сквозного использования

Далее показана последовательность вызовов от загрузки данных до получения
итогового решения; стрелки указывают, какой модуль вызывается на каждом шаге.

1. Создаётся объект `RiskEngine` с конфигурацией. → `config.py`
2. Вызывается метод загрузки длинной истории. → `data/parsers.py` → внутри
   обращается к `data/sources.py`.
3. Полученные цены переводятся в доходности. → `volatility.log_returns`.
4. Запрашиваются оптимальные веса. → `portfolio.py` обращается за матрицей
   ковариаций к `covariance.py`.
5. Запрашивается риск-отчёт по этим весам. → `var_es.py`, опционально
   `evt.py` и `dependence.py`; вызывается `liquidity.bangia_lvar` для оценки
   стоимости закрытия позиции.
6. Запрашивается стрессовая проверка. → `stress.py`.
7. Запрашивается декомпозиция риска. → `var_es.component_var_es`.
8. При поступлении торгового сигнала вызывается метод предторгового
   контроля. → `controls.py` для деривингования по просадке, `liquidity.py`
   для проверки лимита по обороту и расчёта ожидаемого воздействия,
   `var_es.py` для проверки бюджета риска.
9. Возвращается решение о допустимости и размере операции. → `engine.py`.

Каждый из этих шагов может быть вызван по отдельности, что даёт гибкость
интеграции в любую внешнюю систему.
