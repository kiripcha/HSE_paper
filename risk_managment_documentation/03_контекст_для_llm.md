# Контекст для LLM-агента, чтобы продолжить работу

Документ предназначен для агента (или нового разработчика), который
подключается к проекту впервые. Прочтя этот файл, агент должен получить
достаточный объём знаний, чтобы пользоваться пакетом, расширять его и
интегрировать в более крупную систему без чтения первичных источников.

---

## 1. Что это за проект и зачем он

`crypto_risk` — модуль управления рыночными рисками для высокоскоростной
торговой системы, оперирующей на рынке криптовалют. Система использует методы
глубокого обучения для построения торговых сигналов и адаптирует свою работу
к изменяющемуся режиму рынка. Модуль обеспечивает её внутренний контур
управления рисками — оценку, контроль и адаптивное уменьшение позиции при
ухудшении условий.

Модуль состоит из тринадцати компонентов, объединённых в единый оркестратор
`RiskEngine`. Все компоненты используют согласованные форматы данных и могут
вызываться как через оркестратор, так и напрямую, независимо друг от друга.

Не входит в ответственность модуля: построение торгового сигнала, исполнение
заявок в стакан, кредитный и операционный риск, регуляторная отчётность.

## 3. Конвенции, которые надо запомнить

| Конвенция | Описание |
|---|---|
| Знак VaR и ожидаемых потерь | Положительные числа в долях капитала. 0,05 = убыток 5% |
| Аннуализация | Множитель 365 (крипта 24/7), не 252 |
| Базовая котировка | USDT |
| Рыночный индекс | BTC (для расчёта бет) |
| Доходности | Логарифмические; первое наблюдение опускается |
| Веса | Сумма равна 1 для длинных портфелей; могут быть отрицательными при допуске коротких продаж |
| Ковариация | Симметричная положительно полуопределённая |
| Временные индексы | UTC, типа `pandas.DatetimeIndex` |
| Случайное семя | Зафиксировано в `crypto_risk.config.RANDOM_SEED` = 42 |
| Кэш данных | В папке `data_cache/`; разные имена файлов для синтетики и реальных данных |

Эти соглашения позволяют свободно собирать модули в новые конвейеры. Любая
функция, нарушающая их, должна явно обрабатываться вызывающим кодом.

## 4. Минимальный пример полного цикла

Следующий код — единственный, который нужен, чтобы получить полный риск-отчёт
по портфелю криптовалют:

```python
from crypto_risk import RiskEngine, RiskConfig
from crypto_risk.data import TOP10_POPULAR

eng = RiskEngine(RiskConfig(universe=TOP10_POPULAR, capital=1_000_000))
eng.load_data_multi("2015-01-01", "2025-01-01")

weights = eng.optimal_weights(
    objective="max_sharpe",
    constraint="long_only",
    cov_method="shrinkage",
)
report = eng.risk_report(weights=weights, use_copula=True)

print(f"Годовая σ: {report.annualized_vol:.1%}")
print(f"VaR с поправкой на ликвидность: {report.lvar['lvar']:.2%}")
print(report.portfolio_risk.round(4))
```

Если этот код не падает — пакет установлен корректно, окружение собрано
правильно, доступ к источникам данных есть.

## 6. Рецепты типовых задач

### 6.1 Подключить модель глубокого обучения для прогноза волатильности

Достаточно класса с одним методом:

```python
class MyVolatilityNN:
    def predict(self, returns) -> float:
        # вызов обученной нейронной сети
        return predicted_sigma_daily

engine.vol_forecaster.set_dl_model(MyVolatilityNN())
```

После этого все компоненты, которые внутри запрашивают прогноз σ (механизм
целевой волатильности, дробный Келли, стоп-лосс), автоматически используют
нейронную сеть.

### 6.2 Добавить новый источник данных в сборщик

Источник — класс с атрибутом `name` и методом `fetch_close(base, start, end)`,
возвращающим `pandas.Series` с временным индексом в UTC и значениями цены
закрытия в долларах:

```python
class MyExchangeSource:
    name = "my-exchange"
    def fetch_close(self, base, start, end):
        # ... запрос к API
        return pd.Series(values, index=dates, name=base)

from crypto_risk.data import (
    MultiSourceCryptoLoader, CryptoCompareSource, YahooSource,
)
loader = MultiSourceCryptoLoader(
    universe=("BTC", "ETH"),
    sources=[MyExchangeSource(), CryptoCompareSource(), YahooSource()],
)
```

Источники опрашиваются в порядке списка. Первый, вернувший непустой ряд,
становится основным; остальные дозаполняют пропуски.

### 6.3 Использовать собственную ковариационную матрицу для оптимизации

```python
from crypto_risk.portfolio import PortfolioOptimizer

opt = PortfolioOptimizer(my_mu, my_cov, names=my_tickers)
gmv = opt.min_variance(constraint="long_only")
msr = opt.max_sharpe(constraint="long_only")
```

Никаких других зависимостей не требуется — функции оптимизации не привязаны
к остальным частям пакета.

### 6.4 Прогнать обратную проверку собственной модели VaR

```python
from crypto_risk import backtesting as bt
table = bt.run_var_backtests(my_realized_returns, my_var_series, var_alpha=0.99)
print(table)
```

Принимаются произвольные ряды; функция не требует, чтобы они были получены
именно из пакета.

### 6.5 Добавить новый метод оценки VaR

Создать функцию с сигнатурой, совместимой с остальными:

```python
def my_var_es(returns, var_alpha=0.99, es_alpha=0.975, horizon=1):
    var = ...
    es = ...
    from crypto_risk.var_es import RiskEstimate
    return RiskEstimate(
        var=var, es=es,
        method="my_method",
        horizon=horizon,
        var_alpha=var_alpha,
        es_alpha=es_alpha,
    )
```

Для появления в сводной таблице сравнения достаточно дописать вызов в функцию
`var_es.compare_methods`.

### 6.6 Запустить пакет в торговой системе для предторгового контроля

Перед отправкой заявки в стакан:

```python
decision = engine.pre_trade_check(
    asset="BTC",
    notional=signal_size_usdt,
    side="buy",
    adv_usdt=exchange_adv,
    current_drawdown=portfolio.current_drawdown(),
    max_var_budget=0.05,
    max_participation=0.10,
)

if decision.approved:
    place_order(asset="BTC", notional=decision.sized_notional, side="buy")
else:
    log.warning(f"Отклонено: {decision.reasons}")
```

`decision.sized_notional` — фактически разрешённый размер заявки, который
может быть меньше запрошенного.

## 7. Подводные камни, о которых надо знать

* VaR и ожидаемые потери возвращаются ПОЛОЖИТЕЛЬНЫМИ числами. Если внешний
  код ожидает отрицательные — нужно явное преобразование.
* Кэш данных строго различает реальные и синтетические данные через префикс
  имени файла. Удалять оба типа кэшей надо вместе.
* При работе с десятью активами и историей с 2015 года панель доходностей
  содержит пропуски в начале для активов, появившихся позже. Метод
  `.dropna()` сокращает выборку до общего окна (с апреля 2020 для текущей
  вселенной).
* Подгонка модели динамических корреляций занимает заметное время и не
  должна вызываться в горячем цикле. Для быстрых обновлений используется
  выборочная или сжатая ковариация.
* Аннуализация по 365 дней; перевод из 252 (для рынков акций) требуется
  умножением на множитель `sqrt(252/365)`.
* Подгонка GARCH ожидает доходности в долях; внутри функции домножает на 100
  для устойчивости численной оптимизации, затем возвращает результат обратно
  в долях.
* Симуляция через копулу даёт более тяжёлые портфельные хвосты, чем
  параметрические методы; для нагруженного контроля рисков рекомендуется
  использовать именно её.
* Метод наиболее рискованного портфеля решён без использования выпуклой
  оптимизации, поскольку задача невыпукла; в качестве решения берётся
  ближайшая вершина симплекса.

## 8. Зависимости и окружение

Жёстко требуются: `numpy`, `pandas`, `scipy`, `statsmodels`, `scikit-learn`,
`requests`.

Опционально используются: `arch` (для семейства GARCH), `cvxpy` (для
constrained-оптимизации), `ccxt` (для биржевых данных). При отсутствии этих
библиотек соответствующие функции переключаются на фолбэк или возвращают
ошибку с понятным сообщением.

Для запуска ноутбуков нужны: `matplotlib`, `seaborn`, `jupyter`, `nbconvert`,
`ipykernel`.

Зафиксированные точные версии всех зависимостей лежат в файле
`requirements-lock.txt`. Версии с диапазонами — в `requirements.txt`.

Python — 3.12 как основная версия разработки, поддерживается с 3.10.

## 10. Стиль и принципы расширения

* Новые функции пишутся в стиле существующих: с явными сигнатурами, краткой
  внутренней документацией, без избыточных слоёв абстракции.
* Любая новая функция должна быть протестирована — минимально через
  добавление к набору дымовых тестов.
* Не вводятся новые внешние зависимости без явной необходимости. Каждая
  новая зависимость требует обновления `requirements.txt` и
  `requirements-lock.txt`, а также проверки сборки образа Docker.
* Тонкие модули остаются «не знающими друг о друге»; всё новое
  межмодульное взаимодействие добавляется в оркестратор.
* Соглашение о знаке мер риска — нерушимое. Никаких отрицательных VaR не
  должно появляться ни в одной новой функции.

## 12. Что делать, если что-то пошло не так

* Дымовые тесты падают на самом начале (тест данных или волатильности):
  проверить установку базовых зависимостей (`numpy`, `pandas`,
  `statsmodels`).
* Дымовые тесты падают на оптимизации портфеля: проверить установку
  `cvxpy` или использовать альтернативные методы без ограничений.
* Падает на тестах GARCH: проверить установку библиотеки `arch`.
* Падает на тестах копул: проверить версию `scipy` (требуется поддержка
  функций специальных гамма-распределений; именно `scipy.special.gammaln`,
  а не `scipy.stats.loggamma`).
* Падает на загрузке данных: проверить наличие сети или работать в режиме
  `mode="synthetic"`. Для воспроизводимости пакет умеет работать полностью
  оффлайн.
* Сборка образа Docker падает на стадии установки `cvxpy`: проверить, что
  в системе есть `build-essential`, `gfortran` и `libopenblas-dev`.

В случае ошибок, не покрытых перечисленными случаями, идти к коду
соответствующего модуля и читать строку с ошибкой. Все функции снабжены
понятными именами параметров; чаще всего понимание приходит из первой же
страницы.
